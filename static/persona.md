# Professional Interests & Work Persona

## Purpose
This document describes the professional interests, skills, and activities of the person it represents. It is intended for use by an AI agent to score the relevance of a job description on a 0.0–1.0 scale, where 1.0 indicates a near-perfect match and 0.0 indicates no meaningful overlap.

---

## Core Identity

I work across the full lifecycle of data — from understanding why data is needed, through collecting and processing it, to publishing insights and evaluating whether the work delivered value. I am most engaged when data work is purposeful, well-structured, and improves decisions for real users.

---

## Interests by Work Area

### Understanding Needs and Defining Scope
I enjoy working with stakeholders to clarify what information is actually needed and why. This includes identifying gaps in existing data, assessing whether current sources are fit for purpose, evaluating administrative, third-party, or large-scale digital data as alternatives to direct collection, and helping build a clear rationale (business case) for a data initiative before resources are committed.

**Keywords:** stakeholder engagement, user needs analysis, requirements gathering, feasibility assessment, data gap analysis, business case development, scoping, problem definition

---

### Design and Methodology
I am interested in the design of data pipelines and statistical or analytical processes before they are built — defining what variables to collect, how units of analysis are constructed, what methodology will be used to clean and transform data, and how outputs will be structured and disseminated. I value reuse of standards and interoperability.

**Keywords:** methodology design, variable definition, classification schemes, sampling design, survey design, data architecture, workflow design, output specification, standards alignment, metadata schema design

---

### Building Data Systems and Tooling
I enjoy building or configuring the tools and systems needed to collect, process, and publish data. This includes data pipelines, ETL processes, collection instruments, APIs, and dissemination components. I prefer building for reuse and modularity. I value testing — both technical tests and end-to-end process tests before production.

**Keywords:** ETL pipelines, data pipeline development, API integration, tooling, workflow configuration, automated data transfer, collection instruments, system testing, integration testing, modular design, R, Python, SQL

---

### Data Collection and Acquisition
I am interested in how data enters a system — whether through surveys, administrative transfers, API extracts, web scraping, sensor feeds, or file ingestion. This includes frame and sample construction, managing data suppliers, and ensuring that what arrives is what was expected.

**Keywords:** data acquisition, data ingestion, survey operations, administrative data, API data extraction, web scraping, sampling frame, register management, data supplier coordination, multi-source integration

---

### Data Processing and Cleaning
I find data processing deeply satisfying — transforming raw inputs into analysis-ready datasets. This covers validation and editing, handling missing data and imputation, deduplication, record linkage, classification and coding (including machine learning assisted approaches), unit and variable derivation, and aggregation. I care about the logic being transparent and reproducible.

**Keywords:** data cleaning, data validation, edit rules, imputation, record linkage, deduplication, coding, classification, data integration, derived variables, aggregation, reproducibility, data transformation, NLP/ML for data processing

---

### Analysis and Interpretation
I enjoy exploratory and confirmatory analysis — understanding what the data shows, checking estimates for plausibility, identifying anomalies, and producing summary statistics and indicators. I value rigour and transparency in how conclusions are drawn. I am also interested in uncertainty quantification and disclosure control where outputs must protect privacy.

**Keywords:** statistical analysis, exploratory data analysis, data quality review, estimation, anomaly detection, indicator development, uncertainty quantification, statistical disclosure control, confidentiality, time series analysis

---

### Dissemination and Communication
I care about how data reaches its users — structured datasets, publications, dashboards, APIs, open data outputs, and microdata releases. I am interested in discoverability (cataloguing, tagging, metadata), accessibility, and designing outputs that non-specialist users can use effectively.

**Keywords:** data publishing, open data, dashboards, data catalogues, metadata management, API outputs, statistical publications, data visualisation, microdata, discoverability, data documentation

---

### Evaluation and Continuous Improvement
I believe data processes should be evaluated after each cycle — measuring what worked, what didn't, gathering user feedback, and feeding lessons back into the next iteration. I am interested in process metrics, paradata, and using operational data to improve efficiency and quality.

**Keywords:** process evaluation, retrospective analysis, user feedback, process improvement, paradata, operational metrics, lessons learned, quality assessment, continuous improvement

---

## Overarching Interests (Apply Across All Work)

### Quality
I have a strong interest in data quality — not as a one-time check, but as something designed into the process from the start. This includes defining quality criteria, setting targets, monitoring compliance, and reporting against them.

**Keywords:** data quality, quality frameworks, quality assurance, QA/QC, fitness for purpose, quality indicators, error detection

### Metadata
I believe metadata is as important as data itself. I am interested in capturing, managing, and publishing metadata so that data is understandable, traceable, and reusable. This includes provenance, lineage, data dictionaries, and schema documentation.

**Keywords:** metadata management, data lineage, data provenance, data dictionaries, schema documentation, cataloguing

### Data Governance and Management
I am interested in how organisations manage data as an asset — security, access control, retention policies, lifecycle management, and ensuring data is used appropriately and ethically.

**Keywords:** data governance, data stewardship, access control, data security, retention policy, data lifecycle, responsible data use

### Documentation and Knowledge Transfer
I value maintaining clear documentation so processes can be repeated, audited, and improved by others. This includes technical documentation, process guides, and ensuring institutional knowledge is captured.

**Keywords:** technical documentation, process documentation, reproducibility, knowledge management, handbooks, runbooks

---

## Sector Affinity

I am particularly interested in roles that involve public interest data — official statistics, government data, and international organizations working towards the SDGs. However, I am open to any domain where data work is rigorous, purposeful, and operates at scale.

## Scoring Notes for Agents

When scoring a job description against this persona, consider:

- **High relevance (0.7–1.0):** The role involves one or more of the data lifecycle stages described above (design, collection, processing, analysis, dissemination, evaluation), requires rigour in methodology or quality, and involves structured data work at meaningful scale.
- **Medium relevance (0.4–0.6):** The role touches on data work but is primarily focused on adjacent activities (e.g., software engineering without data focus, reporting without analytical depth, or domain expertise without data pipeline work).
- **Low relevance (0.1–0.3):** The role involves incidental data use (e.g., using existing dashboards) but no meaningful contribution to how data is produced, processed, or managed.
- **No relevance (0.0):** The role has no data work component.

Weight most heavily: data processing, methodology, pipeline development, quality, and multi-source data integration. Weight least: generic project management or communication roles with no data specificity.
